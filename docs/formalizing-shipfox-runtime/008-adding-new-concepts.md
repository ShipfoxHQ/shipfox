- Surface, expression, IR, static-semantics, runtime, and durable execution host work are conditional on that classification.

- Defaults, inheritance, expression meaning, and canonical shape changes belong at the normalization/canonicalization boundary before runtime sees the workflow.

## Surface Language Portability Rules

The workflow language should be able to accept future authoring surfaces without making YAML semantics special. Candidate surfaces include CUE, TypeScript SDKs, Python SDKs, generated workflows, and UI builders.

Equivalent workflows from different surfaces should converge before execution:

```text
YAML / CUE / TypeScript SDK / Python SDK / UI Builder
              -> SurfaceWorkflowDocument or canonical normalized model
              -> WorkflowIR
              -> static semantics
              -> runtime
```

Rules:

- Surface syntax may differ; Shipfox semantics must converge.

- Runtime must consume `WorkflowIR` and must not inspect which surface produced it.

- Defaults and inheritance belong to normalization/canonicalization, not to runtime, database persistence, Temporal orchestration, or runner dispatch.

- Expression-bearing fields must eventually parse into shared Shipfox expression IR instead of remaining surface-language strings.

- Diagnostics should point back to the source surface when possible, but diagnostic meaning should be shared across surfaces.

- CUE is a PR1 formalization artifact, not an accepted authoring input. If CUE later becomes canonical, the same semantic rules should move from TypeScript normalization into CUE-backed canonicalization without changing runtime consumption.

## Example: Top-Level Runner Default

A surface may allow a workflow-level runner:

```yaml
runner: ubuntu-latest
jobs:
  build:
    steps:
      - run: npm test
```

This is surface syntax. It is not a runtime rule. If workflow-level `runner` means a default job runner, the default is applied during normalization/canonicalization before producing `WorkflowIR`.

In PR1, that boundary is `normalizeSurfaceDocumentToWorkflowIR` in `libs/api/workflow-language/src/core/ir/normalize-surface-document.ts`:

```text
SurfaceWorkflowDocument
  workflow.runner = "ubuntu-latest"
  jobs.build.runner = undefined

          -> normalization/canonicalization

WorkflowIR
  jobs.build.runner = ["ubuntu-latest"]
```

After this boundary, downstream layers should only see the effective job runner:

```json
{
  "sourceName": "build",
  "runner": ["ubuntu-latest"]
}
```

The runtime, database persistence, Temporal orchestration, and runner dispatch must not re-implement this inheritance rule. They consume the already-normalized effective runner from `WorkflowIR`.

Other surfaces expressing the same intent must normalize to the same effective IR. For example, a TypeScript SDK could expose:

```ts
workflow("simple build")
  .runner("ubuntu-latest")
  .job("build", (job) => job.run("npm test"));
```

A Python SDK could expose:

```python
workflow("simple build") \
  .runner("ubuntu-latest") \
  .job("build", lambda job: job.run("npm test"))
```

Both should normalize to a `WorkflowIR` job with `runner: ["ubuntu-latest"]` before validation, persistence, or execution.
