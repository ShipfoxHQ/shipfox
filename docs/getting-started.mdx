This guide walks you through connecting a repository to Shipfox, writing your first workflow, registering a runner, and firing a run ‚Äî from zero to a live streaming log in under ten minutes. You'll need access to a Shipfox workspace, a Git repository you can connect, and at least one runner online. If you hit a snag at any point, see [Troubleshooting](/troubleshooting).

<Info>
  This guide assumes Shipfox is already running for your team ‚Äî most product engineers join an instance a platform or DevEx team set up. If you need to stand Shipfox up yourself, see [Installation](/installation).
</Info>

## Connect your project

A project connects one repository to Shipfox and is the container for all workflows, runs, and settings related to that repo.

<Steps>
  <Step title="Create a project">
    Open your Shipfox workspace dashboard and click **New project**. Give the project a name that matches the repository you want to connect.
  </Step>
  <Step title="Connect the repository">
    Select the Git provider and choose the repository. Shipfox reads workflow files from `.shipfox/workflows/` in the default branch. No extra configuration is needed ‚Äî any `.yaml` or `.yml` file in that directory is treated as a workflow definition.
  </Step>
</Steps>

## Add your first workflow

A workflow is a YAML file that tells Shipfox when to run (`triggers`), where to run (`runner`), and what to do (`jobs` ‚Üí `steps`). Create the file `.shipfox/workflows/hello.yaml` in your repository with the following content:

```yaml
name: Hello world
runner: ubuntu-latest
triggers:
  on_demand:
    source: manual
    event: fire
  on_push:
    source: github_acme
    event: push
jobs:
  build:
    steps:
      - run: echo "hello world"
```

The fields at a glance:

| Field | What it does |
|---|---|
| `name` | Human-readable name shown in the dashboard and run history. |
| `runner` | Label of the runner that should execute the jobs. Must match a registered runner. |
| `triggers` | A map of named trigger definitions. Each entry has a `source` and an `event`. |
| `jobs` | A map of named jobs. Each job has a `steps` list and optionally a `needs` dependency on other jobs. |
| `steps` | An ordered list of steps. Each step is either a `run` command or an agent step (`model` + `prompt`). |

<Note>
  `filter` is roadmap ‚Äî the `on_push` trigger fires on every push today; branch or path filtering is not yet evaluated. Push triggers require GitHub; GitLab support is on the roadmap. Use a manual trigger (`source: manual, event: fire`) in the meantime if you need local or GitLab testing.
</Note>

Commit and push the file to your connected repository. Shipfox detects the new workflow file automatically ‚Äî no manual import is required.

## Run it

A run is one execution of a workflow. You can start a run in two ways:

- **On demand**: Open the project dashboard, find the workflow, and click **Run now**.
- **On push**: Push a commit to the connected repository ‚Äî the `on_push` trigger fires automatically.

Once the run starts, click into it to stream live output. For the workflow above you should see:

```text
build ¬∑ echo "hello world"
hello world
```

<Note>
  A run stuck in `pending` usually means no registered runner matches the `runner:` label in your workflow. Check that a runner with the label `ubuntu-latest` is online and connected. See [Troubleshooting](/troubleshooting) for steps to diagnose runner connectivity.
</Note>

## Add an agent step

Shipfox treats AI agents as first-class steps. Add an agent step to your workflow by specifying a `model` and a `prompt` alongside your existing `run` step:

```yaml
jobs:
  build:
    steps:
      - run: echo "hello world"
      - name: summarize
        model: claude-opus-4-8
        prompt: Say hello back in one short sentence.
```

Push the updated file. Fire another run and you should see output similar to:

```text
summarize ¬∑ claude-opus-4-8
Hello back! üëã
```

Agent output varies between runs ‚Äî the model's response is non-deterministic, so expect different wording each time. The step is still marked as succeeded or failed based on whether the model call completes without error.

To make the agent *act on your code* ‚Äî fix failing tests, review a diff ‚Äî give it a repo-aware prompt and pair it with a [gate](/concepts/gates) so it retries until a check passes. Agent steps use your workspace's configured [model provider](/reference/model-providers), so configure one before running them.

## Let your agent write workflows

You can use a coding agent to generate and iterate on Shipfox workflows directly in your repository. Because every Shipfox documentation page is available as plain Markdown, your agent can fetch the schema reference and author workflow YAML without any special plugin. Point your agent at the [AI overview](/ai/overview) page or simply describe what you want in natural language ‚Äî for example, "add a Shipfox workflow that runs tests on every push and posts a summary". The agent understands the workflow schema and Shipfox conventions from the docs. See the [AI overview](/ai/overview) for full details on available prompting patterns and context files.
