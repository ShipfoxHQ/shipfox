Shipfox runs **agentic workflows** from your repository. A workflow is a YAML file in `.shipfox/workflows/` тАФ triggers, jobs, steps, runners тАФ a shape you already know how to read. The difference is what a step can be: a step runs a shell command **or runs an AI agent** тАФ a `model`, a `prompt`, and your repository checked out. Agents and commands share one lifecycle, so a gate can retry an agent until your tests pass, and every agent step records a full session: messages, thinking, tool calls, token usage, and cost.

```yaml
name: Fix Sentry issues
runner: ubuntu-latest
triggers:
  on_issue:
    source: sentry_acme            # fires when Sentry reports a new issue
    event: issue.created
jobs:
  fix:
    steps:
      - key: install
        run: npm ci
      - model: claude-opus-4-8
        prompt: >
          Sentry reported "${{ event.title }}" (${{ event.webUrl }}).
          Find the cause in this repository and fix it.
      - run: npm test
        gate:
          success_if: exit_code == 0
          on_failure:
            restart_from: install  # loop until the tests pass
```

If you've written GitHub Actions, you already know how to read this file тАФ that's deliberate. What's different is what a step can do тАФ and the control flow around it: a Sentry event starts the run, the event's data lands in the agent's prompt, and the gate loops the agent until the tests pass. Agents run on any of 30+ model providers with your own API keys, on runners you own. Every run тАФ including the agent's full transcript тАФ streams live to the dashboard.

## How it works

<Steps>
  <Step title="Define a workflow">
    Write a YAML file under `.shipfox/workflows/` in your repository. Commit and push тАФ Shipfox picks it up automatically without any manual import step.
  </Step>
  <Step title="Configure a trigger">
    Choose when the workflow runs: on a GitHub push, a Sentry issue, a webhook from any system, or on demand from the dashboard тАФ with more integrations on the way. Each trigger is a named entry under the `triggers` key in your workflow YAML.
  </Step>
  <Step title="Jobs run on your runners">
    Each job executes on a runner you register with Shipfox. Jobs are fully isolated тАФ each one re-clones the repository before executing, so no state bleeds between runs.
  </Step>
  <Step title="Steps: commands or agents">
    Each job contains one or more steps. A step runs a shell command (`run`) **or an AI agent** тАФ specify a `model` and a `prompt` and it reads your repo, runs tools, and acts. Agent steps are first-class: same lifecycle as shell steps, so they stream output, report status, and can be gated and retried until the result is correct.
  </Step>
  <Step title="Watch it live">
    Open the Shipfox dashboard to stream logs for every run, job, and step in real time. Drill into any step to see its full output, duration, and exit status.
  </Step>
</Steps>

## Shipped vs. roadmap

Features listed as roadmap are not yet available and should not be relied upon in production workflows.
