---
title: "Respond to Pull Request Feedback with a Listening Job"
sidebarTitle: "PR Feedback Listener"
description: "Keep one Shipfox run open for pull-request review comments and resolve it when that pull request closes."
---

This workflow starts when a pull request opens, records its number, and analyzes
later review comments for that pull request until it closes.

## Add the workflow

Create `.shipfox/workflows/pull-request-feedback.yml`:

```yaml
name: Respond to pull request feedback
runner: ubuntu-latest

triggers:
  on_pull_request:
    source: github_acme # Replace with your GitHub connection slug.
    event: pull_request.opened

jobs:
  prepare:
    outputs:
      pr_number: ${{ steps.remember.outputs.pr_number }}
    steps:
      - key: remember
        env:
          PR_NUMBER: "${{ event.number }}"
        run: echo "pr_number=$PR_NUMBER" >> "$SHIPFOX_OUTPUT"
        outputs:
          pr_number: number

  respond:
    needs: prepare
    listening:
      on:
        - source: github_acme # Replace with the same connection slug.
          event: pull_request_review_comment.created
          filter: event.pull_request.number == jobs.prepare.outputs.pr_number
      until:
        - source: github_acme # Replace with the same connection slug.
          event: pull_request.closed
          filter: event.pull_request.number == jobs.prepare.outputs.pr_number
    steps:
      - prompt: >
          Analyze this review comment on pull request
          #${{ jobs.prepare.outputs.pr_number }}:
          ${{ execution.events[0].data.comment.body }}

          Return a proposed response and list any code changes the author
          should make.
```

## Verify the listener

1. Commit and push the workflow.
2. Open a pull request in the connected repository.
3. Open the new Shipfox run and wait for `respond` to show as listening.
4. Add a review comment to that pull request.
5. Confirm that the listener creates an execution containing the comment.
6. Close the pull request and confirm that the listening job resolves.

The filters are required. They prevent comments or close events from another
pull request from changing this run. Batching and resolution options live in the
[Listening fields reference](/reference/workflow-schema#listening-fields).
