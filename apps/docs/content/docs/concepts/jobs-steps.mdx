Every Shipfox workflow run is a directed acyclic graph of **jobs**. A job is a group of **steps** that run in order on a single runner. Steps are either **run steps** (shell commands) or **agent steps** (AI model invocations). Jobs run in parallel by default; `needs` sequences them.

## How a job executes

When a run starts, every job without unmet `needs` is dispatched to a runner whose labels match the job's `runner:` field. The runner clones the repository, executes the steps in order, and streams output back (see [Logging](/concepts/logging)).

<Callout type="warn">
  Each job **re-clones the repository** at the start of its execution. Jobs do not share a filesystem: if job A installs dependencies, job B must install them again. Use `needs` to order jobs, not to pass files between them.
</Callout>

This isolation is intentional. It keeps each job's environment clean and reproducible, and it lets jobs run on different runners without shared-state assumptions.

A job fails when a step fails, and a job-level `execution_timeout` bounds its total wall-clock time. All job fields are in the [Job fields reference](/reference/workflow-schema#job-fields).

## Run steps

A `run` step executes a shell command on the runner: build tools, test suites, linters, any CLI-driven work.

```yaml
jobs:
  test:
    needs: [lint]            # wait for the lint job
    steps:
      - key: install         # key identifies the step (gates can restart from it)
        run: npm ci
      - run: npm test
```

## Agent steps

An agent step invokes an AI model. Instead of `run`, you provide a `prompt`. The model reads the repository context and produces output (edits, reports, suggestions) depending on the prompt.

```yaml
jobs:
  review:
    steps:
      - key: review
        model: claude-opus-4-8
        thinking: high
        prompt: Review the latest diff and summarize any issues.
```

The optional `harness` field picks the agent runtime that executes the step: `pi` (the default) or `claude` (the Claude Agent SDK). The harness determines which `thinking` values the step accepts. When omitted, `model` and `provider` resolve against your workspace's [model provider configuration](/reference/model-providers). All agent-step fields and their accepted values are in the [Agent step fields reference](/reference/workflow-schema#agent-step-fields).

<Callout type="warn">
  `env` is not valid on agent steps, and workflow-level or job-level `env` is not forwarded to them either. See [Environment variables](/reference/workflow-schema#environment-variables).
</Callout>

## Agents that iterate

Pair an agent step with a `run` step that verifies its work, and a [gate](/concepts/gates) that loops back until the check passes:

```yaml
steps:
  - key: fix
    prompt: Fix the failing tests.
  - run: npm test
    gate:
      success: step.exit_code == 0
      on_failure:
        restart_from: fix
```

This is the pattern that turns an agent into an iterating worker. [Gates](/concepts/gates) explains how the loop is evaluated and bounded.
