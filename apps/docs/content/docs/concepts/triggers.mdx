A **trigger** tells Shipfox when to start a workflow run. Triggers are defined in your workflow YAML under the `triggers:` key. You can declare several. Each is a named entry with a `source` (where the event comes from) and an `event` (which event from that source).

For an integration, `source` is the **connection slug** you get after connecting it, not the bare provider name. For on-demand runs it is the literal string `manual`.

## How it works

When an event arrives from a connected integration, Shipfox matches its `(source, event)` pair against every trigger of every workflow in the project. Each match creates one run. The trigger's name (the YAML key you chose) appears in the dashboard and logs, so you can tell which trigger fired a run.

```yaml
triggers:
  on_demand:               # any unique name you choose
    source: manual         # the built-in on-demand source
    event: fire
  on_push:
    source: github_acme    # your GitHub connection slug
    event: push
```

All trigger fields, including the parsed-but-not-yet-evaluated `filter`, are in the [Trigger fields reference](/reference/workflow-schema#trigger-fields). A `source` that matches no connection slug is stored but never fires — see [Trigger Sources](/reference/trigger-sources) for where slugs come from.

## Manual trigger

The simplest trigger needs no integration. It fires when you click **Run** in the dashboard or call the API. A workflow can have at most one manual trigger. Use it for release workflows or one-off automation where you control when the run happens.

## Coming soon

Two trigger capabilities are under active development:

**Cron triggers** — run a workflow on a recurring schedule, declared like any other trigger: `source: cron`, `event: tick`, and a `config` block carrying a 5-field cron `schedule` (minute granularity) and an optional IANA `timezone` (UTC by default). A tick fires the latest definition on the default branch; missed ticks are skipped, not backfilled.

```yaml
# Illustrative — coming soon; syntax may still evolve before release.
triggers:
  nightly:
    source: cron
    event: tick
    config:
      schedule: "0 2 * * *"
      timezone: "Europe/Paris"
```

**Job-level listening** — the same `{source, event}` selector shape, used on a job (`listening.on` / `listening.until`) so it reacts to events repeatedly inside one run. See [Listening Jobs](/concepts/listening-jobs).
