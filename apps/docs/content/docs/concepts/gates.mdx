A **gate** turns a step's outcome into a decision: did this step succeed, and
if not, what should happen? Gates let you build **bounded retry loops** with no
external scripting: run a step, check a condition, and restart from an earlier
step until it passes. This is how Shipfox turns an agent into an iterating
worker. The agent edits, a `run` step judges the result, and the gate sends the
agent back until the check passes.

## How a gate is evaluated

After a gated step finishes, Shipfox evaluates the gate's `success` expression
(CEL, over the step's exit code) and reaches one of three outcomes:

| Outcome | When | Result |
|---|---|---|
| **Passed** | `success` evaluates true | The step succeeds; the job continues |
| **Failed** | `success` is false | Restart from `restart_from` if the budget allows; otherwise the job fails |
| **Uncheckable** | No exit code, or the CEL expression errors | The job fails immediately; never restarts |

Gates are valid on **both run steps and agent steps**. The most reliable loop
still puts the gate on a `run` step that objectively verifies the agent's work
(tests, a linter), with `restart_from` pointing back at the agent.

## A retry loop

```yaml
steps:
  - key: fix
    prompt: Fix the failing tests.
  - run: npm test
    gate:
      success: step.exit_code == 0
      on_failure:
        restart_from: fix
        feedback: Tests are still failing; try again.
```

The gate's fields (`success`, `on_failure.restart_from`, `on_failure.feedback`)
and their rules are in the [Gate fields reference](/reference/workflow-schema#gate-fields).
The expression language is described in [Expressions](/reference/expressions).

## Loop bounds

Gates create loops, so Shipfox bounds them for you. There is no way to author an
unbounded retry.

- **Per-step restart cap.** A gating step gets a bounded number of attempts
  (3 by default; see [Limits](/reference/limits)). When the cap is spent and the
  gate still fails, the step is marked failed and the **whole job stops**.
- **Job execution timeout.** The job-level `execution_timeout` also bounds total
  wall-clock time (6 hours by default; see [Limits](/reference/limits)).
  Whichever bound is reached first ends the loop.

Design loops to converge within a few attempts; the cap is a safety net, not a
budget to spend.
