Shipfox turns YAML files in your repository into agentic automation. A **trigger** (a push, an alert, a manual fire) creates a **run**, and the run schedules **jobs** as a dependency graph. Each job executes **steps** on a **runner**; a step is a shell command or an AI agent. **Gates** loop an agent back until its work passes a check, and everything streams live to the dashboard.

![Run execution flow: event sources match a trigger in the Shipfox control plane, a run is created, jobs are scheduled as a needs DAG, a runner on your compute claims the job and executes run and agent steps with a gate looping back on failure while logs stream live](/img/diagrams/run-execution-flow.png)

**[Workflows](/concepts/workflows)** are YAML files under `.shipfox/workflows/` in your repository: one file, one workflow. They are versioned, diff-able, and reviewed like any other code, and Shipfox picks up changes on push.

**[Triggers](/concepts/triggers)** decide when a run starts. Each trigger names a `source` (an integration connection or `manual`) and an `event` (like `push` or `issue.created`). Any tool that can send an event can start a run.

**[Jobs and steps](/concepts/jobs-steps)** define the work. Jobs run in parallel by default and sequence with `needs`. Each job is an isolated unit that clones the repo fresh. A step is either a `run` shell command or an **agent step**: a model and a prompt, executing with your repository checked out.

**[Gates](/concepts/gates)** make agents iterate. A gate checks a step's outcome with a CEL expression and can restart the job from an earlier step when the check fails. The agent edits, a shell step verifies, and the loop closes when the check passes, within bounded attempts.

**[Logging](/concepts/logging)** shows you everything, live. Run steps stream their output. Agent steps stream structured sessions: messages, thinking, tool calls, token usage, and cost. Everything is persisted for later.

**[Runners](/operations/runners)** execute the jobs. Labels route each job to matching compute. You can bring your own runners for special hardware, private networks, or cost control, and a [provisioner](/operations/runner-provisioners) can autoscale ephemeral runners to demand. Both live in the [Operate](/operations/runners) section.

**[Listening jobs](/concepts/listening-jobs)** (coming soon) extend the model: a job that waits for events inside a run and reacts to each batch, like an agent answering PR review comments until the PR merges.
