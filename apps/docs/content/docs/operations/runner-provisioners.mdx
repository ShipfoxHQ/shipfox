A **provisioner** watches how many jobs are waiting. It starts **ephemeral,
single-job runner containers** to meet that demand, then lets them exit. It is the
alternative to [manually starting runners](/operations/runners): instead of keeping
an idle fleet online, you run one provisioner and it scales runners to the work.

<Callout type="info">
  Today Shipfox ships a **Docker** provisioner. The provisioner core is
  provider-agnostic, so other backends (Kubernetes, cloud VMs) can follow — those
  are roadmap.
</Callout>

## How it works

The provisioner runs a continuous control loop against the Shipfox API:

<Steps>
  <Step>

    **Advertise capacity**

    For each template it can start, the provisioner reports free slots
    (`max_concurrency` minus what's already running).
  </Step>
  <Step>

    **Poll for demand**

    It long-polls the API. When jobs are waiting for labels a template provides, the
    API grants reservations.
  </Step>
  <Step>

    **Mint ephemeral tokens**

    For each reservation, the provisioner mints a single-use **ephemeral
    registration token** (`sf_ert_…`). The new runner uses it to register once.
    It is distinct from the long-lived `sf_mrt_…` manual token; see
    [token types](/reference/runner#token-types).
  </Step>
  <Step>

    **Launch and reconcile**

    It starts a container per reservation. Each container gets the API URL, the
    ephemeral token, and the template's labels. The provisioner reports each
    container's lifecycle, reconciles against Docker on startup, and reaps
    containers that never registered. If Docker is unreachable, it advertises zero
    capacity and backs off instead of double-launching.
  </Step>
</Steps>

Each provisioned runner claims one job, runs it, and exits. A burst of work spins
up runners; quiet periods cost nothing.

## Templates

A **template** maps a label set to the container that provides it. Templates live in
a YAML file the provisioner loads at startup:

```yaml
templates:
  docker-ubuntu22-2vcpu:
    labels: [ubuntu22, ubuntu22-2vcpu]   # labels the started runner registers with
    image: shipfox-runner:ubuntu22       # container image to start
    cpu: 2
    memory: 4GiB
    max_concurrency: 100
```

When several templates satisfy a job's labels, the provisioner picks the cheapest
(fewest vCPUs), then the most specific. The full template schema and every
environment variable are in the
[Runner Provisioner reference](/reference/runner-provisioner).
