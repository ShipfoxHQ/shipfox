Select a provider to review its availability, capabilities, and setup guide.

## Integration connections and slugs

Connecting an integration creates a workspace **integration connection**. Each
integration connection has a stable **slug**. Workflow triggers use the slug as
`source`.

This trigger fragment belongs under an existing workflow's `triggers` map:

```yaml
triggers:
  on_push:
    source: github_acme # A GitHub integration connection slug, not a provider ID.
    event: push
```

A workspace can have several integration connections from one provider. Their slugs keep
their events and agent tools separate.

## Integration trigger contract

All integration triggers use these fields. The [Workflow
Schema](/reference/workflow-schema#trigger-fields) has the full rules.

| Field | Role |
| --- | --- |
| `source` | Integration connection slug that must have delivered the event. |
| `event` | Provider event name recorded by Shipfox. |
| `filter` | Optional CEL rule checked before Shipfox creates a run. |
| `with` | Optional fixed values given to the run as `inputs`. |

The provider pages list the event names and fields that Shipfox owns. They link
to the provider's own guide for raw webhook data.
