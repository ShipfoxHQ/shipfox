Shipfox automates engineering work with **workflows** stored in your
repository. An event starts a run, jobs divide the work, and each job runs
ordered steps on a matching runner.

If you have used GitHub Actions, much of this model is familiar. Shipfox adds
three things:

- A step can ask a coding agent to do work that needs judgment.
- A feedback loop can send work back until a real check passes.
- A listening job can react to outside input without creating another run.

This workflow shows the common path. A Sentry issue starts a run, an agent fixes
the code, and the test suite decides whether the work is ready:

```yaml
name: Fix a Sentry issue
runner: ubuntu-latest

triggers:
  on_issue:
    source: sentry_acme # Replace with your Sentry connection slug.
    event: issue.created

jobs:
  fix:
    steps:
      - key: fix
        prompt: >
          Sentry reported "${{ event.title }}". Reproduce the error in this
          repository and fix the cause.

      - run: npm test
        gate:
          success: step.exit_code == 0
          on_failure:
            restart_from: fix
```

Read it from top to bottom:

- The **trigger** matches an issue from the `sentry_acme` connection and creates
  one run.
- The **job** groups the work and sends it to a runner with the
  `ubuntu-latest` label.
- The **agent step** gets the issue title, the checked-out repository, and a
  goal.
- The **run step** executes the project test command.
- The **gate** checks the exit code. A failed check restarts the agent step. A
  passing check lets the job finish.
