A **feedback loop** lets a job revise its work after another step shows that the
result is not ready. One step does the work, another checks it, and the job goes
back when the result does not pass.

Use a feedback loop when the result can be checked inside the job. Tests, type
checks, linters, scripts, and agent reviews can all act as checks. If the next
attempt must wait for new outside input, use a
[listening job](/understand/listening-jobs) instead.

A **gate** is the decision point attached to the checking step. It uses the
completed step's result. A gate becomes a feedback loop only when its failure
policy sends the job back to an earlier step.

## How a gate is evaluated

A gate evaluates after its step finishes. Its `success` expression reads the
step result and decides whether the check passed.

| Gate result | Restart configured | What happens |
| --- | --- | --- |
| Pass | Either | The job continues to the next step |
| Fail | No | The job fails |
| Fail | Yes | The job restarts from the named earlier step while attempts remain |
| Cannot be evaluated | Either | The job fails without restarting |

This separation matters. A gate can enforce a custom success rule without
creating a loop. The `on_failure.restart_from` policy is what sends the job
back for another pass through the work.

The exact expressions and recovery fields live in the
[gate fields reference](/reference/workflow-schema#gate-fields).

## The loop shape

```yaml
name: Fix failing tests
runner: ubuntu-latest

triggers:
  manual:
    source: manual
    event: fire

jobs:
  fix:
    steps:
      - key: fix
        prompt: >
          Fix the failing tests in this repository.
          Previous check: ${{ step.is_retry
            ? step.restart.feedback
            : "No previous check. Run the tests and inspect the failures." }}

      - key: test
        run: npm test
        gate:
          success: step.exit_code == 0
          on_failure:
            restart_from: fix
            feedback: The tests still fail. Run them again, inspect the failures, and fix the cause.
```

The `fix` step changes the repository. The `test` step checks the result, and
the gate treats exit code `0` as a pass. A failure restarts the job from `fix`,
then the test runs again.

The prompt checks `step.is_retry` before reading the gate feedback. The first
attempt gets its initial instruction. Later attempts get the reason the check
sent the job back.

## Feedback guides the next attempt

Trying the work again without new information often repeats the same mistake.
Gate feedback carries the reason the job went back. The earlier step reads it
from `step.restart.feedback`.

Feedback can be fixed guidance, as in the example, or include outputs from the
checking step. Give the next attempt the evidence it needs: a failed test, a
review finding, or the missing condition that kept the result from passing.

[Data and templating](/understand/data-and-templating) explains how step outputs
can feed that message.

## Loop bounds

Feedback loops stop when the gate passes or the restart budget is exhausted.
An execution timeout can also stop the whole job. If the result never passes,
the job fails rather than running forever.

These bounds are a safety net. A loop that often reaches them usually needs a
clearer prompt, a better check, or a smaller task. The exact values live in
[Limits](/reference/limits).

## What the run shows

Each attempt appears in the run with its step result, gate decision, restart
point, and feedback. The run keeps the whole cycle as structured history
instead of hiding it inside one long retry log.
