No matter which coding agent you use, you can get valid Shipfox workflow YAML by pasting this prompt into your session. It contains the complete shipped schema, the hard validation rules, the today/roadmap boundary, and two correct examples; the same content as the skills package, in a single paste.

## The prompt

```text Copy this into your coding agent
You are writing a Shipfox workflow (YAML under .shipfox/workflows/). A trigger
starts a run; a run is a DAG of jobs; each job runs steps on a runner. Output only
valid YAML.

Top level: name (required), runner (label or list), env (run-step only), triggers,
jobs (required). Unknown keys are rejected.

Trigger { source, event, filter?, with? }. For integrations, source is the
CONNECTION SLUG (e.g. github_<org>, sentry_<org>, or your webhook slug), not the
bare provider name; for on-demand runs it is literally "manual". Shipped events:
manual/fire, github push plus any other GitHub webhook as <event>.<action>
(e.g. pull_request_review_comment.created), sentry issue.(created|resolved|
assigned|archived|unresolved), custom-webhook/received. filter is CEL over the
event payload, evaluated when the event arrives (false skips it; an error fails
closed). At most one manual trigger, and manual triggers cannot have filter.
with values surface to steps as ${{ inputs.* }}.

Job { steps (>=1), needs?, runner?, env?, if?, outputs?, execution_timeout?,
listening?, name? }. Jobs run in parallel; each is isolated and re-clones the
repo (install deps per job). if is exactly one ${{ }}-wrapped CEL boolean;
false marks the job skipped. Job outputs values are ${{ }} templates over
steps.<key>.outputs. listening { on (trigger[]), until? (trigger[]), timeout?,
max_executions?, batch? {debounce, max_size, max_wait}, on_resolve?
(finish|cancel) } makes the job fire one execution per matching event batch;
it must declare at least one of until/timeout/max_executions.

Step is a run step { run, key?, name?, if?, env?, outputs?, gate? } OR an agent
step { prompt, model?, harness?, thinking?, provider?, key?, name?, if?,
outputs?, gate? }; never both; agent requires prompt; env is run-only. key
identifies a step; name is display-only. harness is "pi" (default) or "claude".
thinking depends on the harness (pi: off|minimal|low|medium|high|xhigh; claude:
low|medium|high|xhigh|max; default xhigh).

outputs declares typed values, e.g. outputs: { count: number, report: json }.
A run step emits key=value lines to the file named by $SHIPFOX_OUTPUT; an agent
step calls its set_output tool. Later expressions read
${{ steps.<key>.outputs.<name> }} and ${{ needs.<job>.outputs.<name> }}.

gate (run or agent steps) { success?: "<CEL over step.exit_code, step.status,
step.outputs>", on_failure?: { restart_from: "<earlier step's key, NOT its
name>", feedback?: "<message, may use ${{ }}>" } }. Example: success:
step.exit_code == 0. The restarted step reads ${{ step.restart.feedback }}.
Restarts are capped (3 attempts per gating step) before the job fails.

${{ }} templates work in run/env values/prompt/model/provider/names/job
outputs/gate feedback, filled from run/trigger/event/inputs/vars/steps/needs
context as each becomes available. ${{ vars.KEY }} is workspace config, usable
in any template site. ${{ secrets.KEY }} is usable ONLY in a run step's run or
env; never in prompts or predicates. Predicates (filter/if/gate.success/job
success) cannot read vars or secrets. Do NOT use (not shipped): loop, matrix,
branch. event.ref on a push is the full Git ref (e.g. "refs/heads/main").
```

## How to use it

<Steps>
  <Step>

    **Copy the prompt**

    Click the copy button on the prompt above to copy the full schema cheat sheet to your clipboard.
  </Step>
  <Step>

    **Paste into your session**

    Paste at the start of a new conversation in Claude, ChatGPT, Cursor, or your preferred coding agent tool.
  </Step>
  <Step>

    **Ask for your workflow**

    Describe what you want, for example: "Add a Shipfox workflow that runs my tests on every GitHub push and then asks Claude to review the results."
  </Step>
  <Step>

    **Place the output**

    Save the YAML to `.shipfox/workflows/<name>.yml` in your repo and commit it.
  </Step>
</Steps>

## What the prompt covers

The prompt is deliberately compact but complete. It covers every dimension of the shipped schema:

- **Top-level schema fields**: `name`, `runner`, `env`, `triggers`, `jobs`, and which keys are rejected
- **Trigger sources (shipped today)**: `manual`, plus integration connection slugs for GitHub (`push` and other events), Sentry (`issue.*`), and custom webhooks (`received`), with evaluated CEL `filter`s
- **Job isolation, `needs`, `if`, and `outputs`**: parallel execution, repo re-cloning, conditionals, and data passing
- **Listening jobs**: the `listening` block that fires one execution per event batch until resolved
- **Step kinds: run vs. agent**: the two mutually exclusive step shapes, their allowed fields, and typed step `outputs`
- **Gate / restart_from rules**: CEL over exit code, status, and outputs, plus restart feedback
- **Data rules**: `${{ vars.* }}` everywhere, `${{ secrets.* }}` in run-step run/env only, no vars or secrets in predicates
- **The today/roadmap boundary**: an explicit list of features not to use (`loop`, `matrix`, `branch`), so the agent doesn't hallucinate roadmap keys

## Example exchange

Here is a sample user prompt and the YAML a well-prompted agent should produce.

**User prompt:** "Add a Shipfox workflow that runs npm test on every push and has an agent step summarize failures."

**Expected output:**

```yaml
name: Test and triage
runner: ubuntu-latest
triggers:
  on_push:
    source: github_acme    # your GitHub connection slug
    event: push
jobs:
  test:
    steps:
      - run: npm ci
      - run: npm test
  triage:
    needs: test
    steps:
      - model: claude-opus-4-8
        thinking: high
        prompt: If the tests failed, summarize the failure and suggest a fix.
```
