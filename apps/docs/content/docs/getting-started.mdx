This guide walks you through connecting a repository to Shipfox, writing your first workflow, firing a run, and ending with an agent that fixes code in a self-correcting loop.

<Callout type="info">
  This guide assumes Shipfox is already set up for your team: you can sign in to a workspace and at least one runner is online. If you need to stand Shipfox up yourself first, see [Installation](/installation); to add runners, see [Operations](/operations/runners).
</Callout>

## Connect your project

A project connects one repository to Shipfox and is the container for all workflows, runs, and settings related to that repo.

<Steps>
  <Step>

    **Create a project**

    Open your Shipfox workspace dashboard and create a project. Pick the source connection, then the repository. The project name is pre-filled from the repository and can be edited.
  </Step>
  <Step>

    **Connect the repository**

    Shipfox reads workflow files from `.shipfox/workflows/` in the default branch. No extra configuration is needed: any `.yaml` or `.yml` file in that directory is treated as a workflow definition.
  </Step>
</Steps>

## Add your first workflow

A workflow is a YAML file that tells Shipfox when to run (`triggers`), where to run (`runner`), and what to do (`jobs` → `steps`). Create the file `.shipfox/workflows/hello.yaml` in your repository with the following content:

```yaml
name: Hello world
runner: ubuntu-latest      # label of the runner that executes the jobs
triggers:
  on_demand:
    source: manual         # fires when you click Run in the dashboard
    event: fire
jobs:
  build:
    steps:
      - run: echo "hello world"
```

The manual trigger works in every workspace with no further setup. Every field is described in the [Workflow Schema reference](/reference/workflow-schema).

Commit and push the file to your connected repository. Shipfox detects the new workflow file automatically. No manual import is required.

## Run it

A run is one execution of a workflow. Open the project's **Workflows** page, find the workflow, and click **Run**.

Once the run starts, click into it to stream live output. For the workflow above you should see:

```text
build · echo "hello world"
hello world
```

![Shipfox run detail page showing a succeeded run with its job graph and step list](/img/run-detail.png)

*A run detail page: status, the job graph, and every step with its duration.*

<Callout type="info">
  A run stuck in `pending` means no online runner matches the `runner:` label in your workflow. Check that a runner with the label `ubuntu-latest` is online. See [Runners](/operations/runners#pending-runs).
</Callout>

## Make it automatic

To fire the workflow on every push, add a second trigger whose `source` is your GitHub **connection slug**, not the provider name. Find your slugs on the [integrations settings page](/integrations/overview#finding-your-connection-slug); they look like `github_<org>`.

```yaml
triggers:
  on_demand:
    source: manual
    event: fire
  on_push:
    source: github_acme    # replace with YOUR connection slug
    event: push
```

<Callout type="info">
  Push triggers require GitHub today. GitLab is on the roadmap. To run only on
  some branches, add a `filter` such as `event.ref == "refs/heads/main"`. See
  [Trigger Sources](/reference/trigger-sources#filter).
</Callout>

## Add an agent step

Shipfox treats AI agents as first-class steps. Instead of `run`, give a step a `prompt` (and optionally a `model`).

<Callout type="warn">
  Agent steps use your workspace's configured [model provider](/reference/model-providers). Configure one first in **Settings → Agents**. An API key is enough. Without a configured provider, the step fails at execution time.
</Callout>

```yaml
jobs:
  build:
    steps:
      - run: echo "hello world"
      - key: explain
        model: claude-opus-4-8
        prompt: >
          Read this repository and explain in two sentences what it does
          and how you would run its tests.
```

Push the updated file and fire another run. You should see output similar to:

```text
explain · claude-opus-4-8
This repository is a small demo service... To run its tests, ...
```

The agent had your repository checked out and read real files to answer. Open the step in the dashboard to see the full **agent session**: every message, thinking block, and tool call, plus token usage and cost. Output varies between runs. The step still succeeds or fails based on whether the call completes.

## Watch it fix itself

The step above only reads. The pattern that makes Shipfox different: pair an agent that **acts** with a `run` step that **verifies**, and a [feedback loop](/understand/feedback-loops) that sends the agent back until the check passes. Add this job (it assumes an `npm test` project; adapt the check to your stack):

```yaml
jobs:
  fix:
    steps:
      - key: edit
        model: claude-opus-4-8
        prompt: The tests are failing. Edit the source code to fix them.
      - run: npm test
        gate:
          success: step.exit_code == 0
          on_failure:
            restart_from: edit   # the agent tries again until tests pass
```

Fire the run and watch the loop. The agent edits, `npm test` judges, and the gate restarts the agent until the tests go green (bounded automatically; see [Limits](/reference/limits)). You just built a self-correcting workflow with twelve lines of YAML.

## Let your agent write workflows

You can use a coding agent to generate and iterate on Shipfox workflows directly in your repository. Because every Shipfox documentation page is available as plain Markdown, your agent can fetch the schema reference and author workflow YAML without any special plugin. Point your agent at the [AI overview](/ai/overview) page or simply describe what you want in natural language, for example "add a Shipfox workflow that runs tests on every push and posts a summary".
